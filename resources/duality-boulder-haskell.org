* So, what‚Äôs duality?

  :speakernotes:
A little CT to start with: for any category /C/, there is an opposite (or dual) category C^op with all the arrows reversed (but the same objects). It follows from reversing the arrows again, that the dual category of C^op is C.

So far, so easy, right? That‚Äôs fundamentally all there is to it, but there are some complications
  :END:      

- C
- C ‚Ü¶ C^op
- (C^op)^op = C

** What does it mean to ‚Äúreverse the arrows‚Äù?

   :speakernotes: 
This is when we start to get into practical complications. Let‚Äôs look at a simple case ~a -> b~. An arrow is /sort of/ another name for function, so the dual of ~a -> b~ is ~b -> a~. Ok, great. Now let‚Äôs look at something a bit more complicated:
   :END:      

#+begin_src haskell
class Functor f where
  fmap :: (a -> b) -> f a -> f b
#+end_src















*** Reversed!

    :speakernotes: 
What? Didn‚Äôt we just look at this? Yeah, ok, but it was for a different reason ‚Ä¶ sort of. So let‚Äôs reverse the arrows of ~Functor~.
    :END:

#+begin_src haskell
class Functor f where
  fmap :: (a -> b) -> f a -> f b


cofmap :: (a <- b) <- f a <- f b

cofmap :: f b -> f a -> (b -> a)
#+end_src

:speakernotes: 
So, something isn‚Äôt right here ‚Äì there‚Äôs no way you can have a function that takes an ~f b~ and and ~f a~ and returns a function of ~b -> a~. It turns out the arrows are /different/. But let‚Äôs go back to our ‚Äúcategorical‚Äù ~Functor~ from earlier:
:END:


*** Categorical
      
#+begin_src haskell
class (Category (Cat j), Category (Cat k)) =>
      Functor (f :: j -> k) where
  fmap :: Cat j a b -> Cat k (f a) (f b)
#+end_src

    :speakernotes: 
With this definition we can now talk about ~Cat j a b~ ‚Äúa arrow in category j from a to b‚Äù. Well, clearly in this case ‚Äúreversing the arrows‚Äù is obvious:
    :END:
      
#+begin_src haskell
  cofmap :: Cat j b a -> Cat k (f b) (f a)
#+end_src

#+begin_src haskell
  -- Cat Type = (->)
  cofmap :: (b -> a) -> f b -> f a
#+end_src

    :speakernotes: 
And this leads to another revelation ‚Äì Functors are self-dual. You can see that just swapping the variables ~a~ and ~b~ gives us a ~cofmap~ that is equvalent to the original ~fmap~.
    :END:

*** Takeaways      

- The objects of a category and its dual are the same
- The arrows of a category ond its dual are the same, but reversed
- ‚à¥ ~Cat j~ is sort-of its own dual, and ~Cat j a b~ is dual to ~Cat j b a~
- ‚à¥ ~Functor~ is its own dual and ~fmap~ is its own dual

#+begin_src haskell
type Op c a b = Cat c b a
#+end_src

:speakernotes: 
which lets us refer to the opposite category, and now we can say that the dual of ~Cat~ is ~Op~ ‚Ä¶ which is great as long as we‚Äôre dealing with ~Cat~, but we rarely get to do that in production Haskell code, right? I mean, it kills me, but it‚Äôs true.
This is something we end up doing a lot ‚Äì generalizing some Haskell thing to it‚Äôs CT equivalent, figuring out the duals, and then applying them back to the Haskell thing.
:END:

** Notions similar to duality

*** Variance

    :speakernotes:
One expectation that people often have is that the dual of `Functor` (aka, a ‚Äúcovariant‚Äù functor) must be a contravariant functor.
    :END:


#+begin_src haskell
class Functor f where
  fmap :: (a -> b) -> f a -> f b

class Contravariant f where
  contramap :: (b -> a) -> f a -> f b
#+end_src

:speakernotes:
In other languages you may see variance in different contexts ‚Äì¬†in Scala, you see it on type parameters, like ~Foo[+A]~ or ~Foo[-A]~ this is the same concept.

You can see here that the mapping function has been reversed. But to be a dual, what else needs to happen? That‚Äôs right ‚Äì¬†the resulting function also needs to be in its opposite category ‚Ä¶ which would give us back a regular /covariant/ ~Functor~ again.

Now ‚Ä¶ what happens if we do take the dual of ~Contravariant~?
:END:


#+begin_src haskell
class Cocontravariant f where
  cocontramap :: (a -> b) -> f b -> f a
#+end_src

    :speakernotes: 
We‚Äôre right back where we started ‚Äì so ~Contravariant~ is also self-dual.

How about /invariant/ functors?
    :END:
      

#+begin_src haskell
class Invariant f where
  imap :: (a -> b) -> (b -> a) -> f a -> f b

class Coinvariant f where
  coimap :: (b -> a) -> (a -> b) -> f b -> f a
#+end_src

    :speakernotes: 
Ok, perhaps you‚Äôre unsurprised at this point. If so, great! That means you‚Äôre getting a bit of a sense of this.

Looking at this, we can see that variance is independent of duality.

* Is it useful to know the dual of something?















  :speakernotes:
That is a /really/ good question. If we know the dual of something, what can we do with it? Not much, really.

However, it is useful to be able to /construct/ the dual of something. There are non-zero instances where I have ‚Äúsimply‚Äù reversed the arrows for something and discovered some other structure that was also useful to me.

This actually happened last weekend. On Friday, [[https://www.reddit.com/r/haskell/comments/7oav51/i_made_a_monad_that_i_havent_seen_before_and_i/][someone discovered a new monad]]. Without getting into the details, if you constrain the ~Store~ comonad with a monoid, you can get a monad instance out of it. That‚Äôs already pretty cool. But then Ed Kmett said ‚Äúif you can get a monad from ~Store~, then you can probably get a comonad (the dual of a monad) from ~State~ (the dual of ~Store~).‚Äù Then [[http://comonad.com/reader/2018/the-state-comonad/][he did it]]. And he noticed that this was already something he was using concretely: ~extract = flip evalState mempty~ is something that comes up a lot when dealing with state already, but Ed also had a more constrained comonad defined for streaming.

As an aside, Kmett has a ~newtype Co w a~ that converts any comonad into its dual monad, but the reverse isn‚Äôt possible within Haskell, so he had to manually construct ~Comonad State~ and from there can /hopefully/ verify that ~Co (State s)~ is equivalent to ~Store s~.

And, it turns out that even though /Haskell isn‚Äôt category theory/, we can still encode the rules of duality well enough to have Haskell do the work of finding a dual for us (at least in a lot of cases). Which means we can now just /ask/ for the dual of something, and Haskell will tell us. And then we have a new structure ‚Ä¶ which /may/ be useful.

Not to mention that if you‚Äôre working in a domain that deals with duals a lot (as I do, with recursion schemes), you can avoid writing a lot of code in this way.
  :END:

** Ergonomics

   :speakernotes: 
I‚Äôm going to jump right to the fun part here.
   :END:
      

#+begin_src haskell
data Fix f = Fix { project :: f (Fix f) }
labelSelfDual ''Fix -- Not actually true
labelDual 'Fix 'project

makeDualDec
  [|d cata :: (f a -> a) -> Fix f -> a
      cata f = f . fmap (cata f) . project |]
  "ana"
#+end_src

:speakernotes: 
So, we declare a new data type representing the fixed-point of functors. We don‚Äôt need to care about the specific data type so much, other than to note that it is its own dual, and that the constructor for it is dual to its sole accessor. Then we define a function that uses this type, wrapping it in a bit of magic that says ‚Äúalso define the dual of this function, and give it the name ~ana~.‚Äù The generated dual looks like:
:END:      
      

#+begin_src haskell
ana :: (a -> f a) -> a -> Fix f
ana f = Fix . fmap (ana f) . f
#+end_src

:speakernotes: 
So, let‚Äôs walk through this a bit ‚Ä¶ the first argument is a function (and the dual of a function just inverts the arrows, as we‚Äôve seen). And then we hit the same issue we did with ~Functor~ ‚Äì rather than this being a function of two arguments, it‚Äôs really a function that takes a function and returns a function. So, we flip the arrow of the returned function as well. So, we now have the correct type for ~ana~.

Now, the value ‚Ä¶ first, the dual of ~f . g~ is ~g . f~, so all the compositions should be flipped, which means we compare ~project~ to ~Fix~, ~fmap~ to ~fmap~, and ~f~ to ~f~. We‚Äôve just declared that ~Fix~ and ~project~ are duals, so that works. Variables are sort of self-dual (their types have been dualized, so it really represents a different expression), so ~f~ works as well. Finally, we have ~fmap~. We know that that is self-dual already, but we have to take the dual of the function we pass to it, and as we‚Äôve just defined here, ~ana~ is (going to be) the dual of ~cata~, so we swap that in there, and great ‚Äì it all checks out!
:END:
